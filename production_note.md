# ข้อควรระวังเมื่อนำระบบขึ้นใช้งานจริง (Production Deployment Notes)

จากการตั้งค่าความปลอดภัยและการเชื่อมต่อระบบกับ FortiGate Captive Portal มี 3 จุดสำคัญที่ต้องระวังและตรวจสอบเมื่อนำระบบขึ้นใช้งานจริงบน Production เพราะอาจส่งผลให้ผู้ใช้งานไม่สามารถล็อกอินหรือใช้งานระบบได้

---

## 1. การตั้งค่า Cookie ผ่าน HTTPS เท่านั้น (`session.cookie_secure = 1`)

**🔥 ระดับความอันตราย: สูงมาก (ล็อกอินไม่ได้เลย)**
- **ปัญหา**: การตั้งค่านี้ใน `security_config.php` จะบังคับให้เบราว์เซอร์ส่ง Session Cookie เฉพาะเมื่อรันผ่านโปรโตคอล `HTTPS` เท่านั้น 
- **ผลกระทบ**: หากเซิร์ฟเวอร์ของคุณถูกเข้าถึงผ่าน `http://` ธรรมดา (เช่น การทดสอบภายใน, เรียกผ่าน IP โดยตรง, หรือ Reverse Proxy ฝั่งในส่งมาเป็น HTTP) เบราว์เซอร์จะไม่ยอมจำ Session ส่งผลให้ผู้ใช้ล็อกอินสำเร็จแต่พอระบบเปลี่ยนหน้าไป Session จะว่างเปล่า และถูกเตะออกจากระบบทันที
- **วิธีตรวจสอบและแก้ไข**: 
  - ต้องมั่นใจว่าโดเมนและ URL ที่ FortiGate ทำการ Redirect ผู้ใช้มาหา (พารามิเตอร์ URL) เป็น `https://...` เสมอ
  - หากจำเป็นต้องรันบน `http://` ธรรมดาจริงๆ ให้เข้าไปเปลี่ยนค่า `ini_set('session.cookie_secure', 1);` เป็น `0` ในไฟล์ `security_config.php`

---

## 2. นโยบาย X-Frame-Options (Clickjacking Protection)

**🔥 ระดับความอันตราย: ปานกลาง (หน้าจอไม่แสดงผล)**
- **ปัญหา**: ในไฟล์ `.htaccess` มีคำสั่ง `Header always set X-Frame-Options "SAMEORIGIN"` ซึ่งทำหน้าที่บล็อกไม่ให้เว็บอื่นๆ ดึงหน้าเว็บเราไปแสดงผลใน `<iframe>`
- **ผลกระทบ**: อุปกรณ์ FortiGate หรือ Captive Portal บางรุ่น เวลาเด้งหน้าต่างขึ้นมาบนมือถือ อาจใช้เทคนิคดึงหน้าเว็บเราไปฝังใน iframe ของหน้า Portal หากเป็นเช่นนั้น หน้าล็อกอินของคุณจะเป็น **หน้าจอขาวเปล่าๆ หรือขึ้น Connection Refused** ทันที เพราะถูกเบราว์เซอร์บล็อกไว้
- **วิธีตรวจสอบและแก้ไข**: 
  - หากทดสอบต่อ Wi-Fi แล้วหน้าต่าง Captive Portal ไม่โหลดหน้าเว็บ ให้เข้าไปลบบรรทัด `Header always set X-Frame-Options "SAMEORIGIN"` ออกจากไฟล์ `.htaccess`

---

## 3. การซ่อน Error เพื่อความปลอดภัย (`display_errors = 0`)

**🔥 ระดับความอันตราย: ต่ำ (มีผลกับ User Experience และการ Debug)**
- **ปัญหา**: เพื่อไม่ให้โครงสร้างระบบรั่วไหล ระบบถูกตั้งค่าไม่ให้แสดง Error Message ออกทางหน้าจอ
- **ผลกระทบ**: หากฐานข้อมูล Radius ล่ม, รหัสผ่านฐานข้อมูลผิด, หรือ API ภายนอกมีปัญหา ผู้ใช้งานจะเห็นแค่ **หน้าจอสีขาว (White Screen of Death)** โดยไม่มีข้อความแจ้งเตือนใดๆ
- **วิธีตรวจสอบและแก้ไข**:
  - หากผู้ใช้งานแจ้งว่าหน้าจอขาว หรือระบบไม่ทำงาน ให้ผู้ดูแลระบบเข้าไปดูไฟล์ **`error.log`** (`/var/www/api/error.log`) เพื่อหาสาเหตุของปัญหาแทนการดูหน้าจอ
  - ระบบได้จำกัดสิทธิ์ (chmod 666) ให้ไฟล์ `error.log` สามารถถูกเขียนโดย Web Server ได้แล้ว
